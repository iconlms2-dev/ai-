# 에이전틱 아키텍처 설계 검증 요청

## 내 상황

나는 이커머스(건강기능식품) 1인 사업자이고, 마케팅 자동화 프로그램을 직접 만들어 쓰고 있다. 현재 Python FastAPI 백엔드(7,700줄) + HTML 프론트엔드(7,089줄) 모놀리식 구조이고, 네이버 블로그/카페/지식인, 유튜브, 틱톡, 쓰레드 등 10개 채널에 콘텐츠를 자동 생성하는 시스템이다.

## 내 목표

**최종 목표**: 디스코드/슬랙으로 AI 직원들에게 지시하고, 결과물만 받아서 최종 판단/승인하는 구조. 진짜 회사의 사장처럼 운영하고 싶다.

- AI가 각 분야의 전문가처럼 동작
- 지시 → AI가 알아서 실행 → 결과 보고 → 내가 승인
- 콘텐츠 퀄리티가 최우선 (피드백 루프로 자동 개선)
- 개발도 AI가 직접 수행 (코드 수정, 테스트, 배포)
- 실수가 반복 불가능한 구조 (하네스)

## 현재 기술 스택

- **백엔드**: Python, FastAPI (port 8000)
- **프론트엔드**: Vanilla HTML/CSS/JS
- **AI**: Claude API (Anthropic), Gemini API (Google)
- **연동**: Notion API, 네이버 검색광고 API, Cafe24, ElevenLabs TTS, Meta Threads
- **개발 도구**: Claude Code (Anthropic의 AI 코딩 에이전트)

## 적용하려는 에이전틱 엔지니어링 5요소

1. **Agent & Sub-Agent**: 멀티 에이전트 오케스트레이션
2. **Context Engineering**: 3레이어 컨텍스트 (시스템/대화/도구)
3. **Tool & MCP**: 도구 계층 구조
4. **Hook & Skill**: 자동 인터셉터 + 베스트 프랙티스 문서
5. **Planning & HITL**: 계획 수립 + 핵심 포인트에서 사람 확인

그리고 **하네스**(실패가 구조적으로 반복 불가능하도록 구조 자체를 고치는 것)를 적용한다:
- 컨텍스트 파일 (CLAUDE.md = 강제 규칙)
- CI/CD 게이트 (자동 검사)
- 명시적 도구 경계 (읽기전용/쓰기전용)
- 지속적 피드백 루프 (실수→규칙추가→하네스강화)

## 설계한 아키텍처

### 전체 구조

```
[나 (사장)] ─── 디스코드/슬랙 또는 Claude Code CLI
    │
    ▼
[메인 세션 = Claude Code]
  CLAUDE.md가 하네스(강제 규칙) 역할
  - 운영 모드: 파이프라인 에이전트를 spawn하여 콘텐츠 생성
  - 개발 모드: 직접 코드 수정 + 테스트 + 배포
    │
    ├→ [shorts-pipeline] → data-researcher → strategist → script-writer → script-reviewer → tts-converter → notion-saver
    ├→ [blog-pipeline]   → keyword-analyzer → strategist → title-generator → script-writer → script-reviewer → notion-saver
    ├→ [cafe-seo-pipeline] → ...
    ├→ [cafe-viral-pipeline] → ...
    ├→ [jisikin-pipeline] → ...
    ├→ [youtube-pipeline] → ...
    ├→ [tiktok-pipeline] → ...
    ├→ [community-pipeline] → ...
    ├→ [powercontent-pipeline] → ...
    └→ [threads-pipeline] → ...
```

**별도 총괄 오케스트레이터 에이전트는 만들지 않는다.** 메인 세션(Claude Code) + CLAUDE.md(하네스)가 총괄 역할을 한다. 이유: 별도 에이전트를 거치면 한 단계 더 느려지고, 메인 세션이 이미 라우팅 판단을 할 수 있다.

### 에이전트 계층 (2단)

**L1 파이프라인 에이전트 (10개)** — 채널별 오케스트레이터:
| 파이프라인 | 단계 |
|-----------|------|
| shorts-pipeline | 벤치마킹 → 전략 → 대본 → 품질검수(→리비전) → TTS → 저장 |
| blog-pipeline | 키워드분석 → 전략 → 제목 → 본문 → 품질검수(→리비전) → 저장 |
| cafe-seo-pipeline | 키워드분석 → 전략 → 제목+본문 → 댓글 → 품질검수 → 저장 |
| cafe-viral-pipeline | 1단계(관심) → 2단계(문제) → 3단계(솔루션) → 품질검수 → 저장 |
| jisikin-pipeline | 키워드분석 → 질문생성 → 답변생성 → 품질검수 → 저장 |
| youtube-pipeline | 영상검색 → 영상분석 → 댓글생성 → 품질검수 → (자동게시) → 저장 |
| tiktok-pipeline | 키워드분석 → 스크립트생성 → 품질검수 → 저장 |
| community-pipeline | 전략수립 → 게시글생성 → 댓글생성 → 품질검수 → 저장 |
| powercontent-pipeline | 레퍼런스수집 → 분석 → 광고카피 → 본문 → 품질검수 → 저장 |
| threads-pipeline | 레퍼런스수집 → 유형선택 → 콘텐츠생성 → 품질검수 → 발행 → 저장 |

**L2 서브에이전트 (15개)** — 전문 작업자, 채널 간 재사용:

| 서브에이전트 | 역할 | 도구 경계 | 사용하는 파이프라인 수 |
|-------------|------|----------|---------------------|
| data-researcher | 외부 데이터 수집 | 읽기전용 | 4개 |
| pattern-extractor | 패턴/팩트 추출 | 읽기전용 | 2개 |
| keyword-analyzer | SERP분석+경쟁강도 | 읽기전용 | 4개 |
| video-analyst | 영상 정보 분석 | 읽기전용 | 2개 |
| strategist | 전략/컨셉 수립 | 생성전용 | 5개 |
| hook-designer | 훅/CTR 설계 | 생성전용 | 2개 |
| title-generator | 제목 생성 | 생성전용 | 3개 |
| **script-writer** | **본문/대본 작성** | **생성전용** | **9개 (거의 전체)** |
| comment-writer | 댓글 생성 | 생성전용 | 4개 |
| **script-reviewer** | **품질검수 (규칙+AI)** | **읽기전용** | **10개 (전체)** |
| forbidden-checker | 금칙어 체크 | 읽기전용 | 3개 |
| tts-converter | TTS+자막 변환 | 특수도구 | 1개 |
| **notion-saver** | **Notion DB 저장** | **쓰기전용** | **10개 (전체)** |
| youtube-poster | 유튜브 게시 | 쓰기전용(HITL) | 1개 |
| threads-publisher | 쓰레드 발행 | 쓰기전용(HITL) | 1개 |

### 스킬 구조 (commands/)

사용자가 `/명령어`로 직접 호출하는 트리거:
- 채널 트리거 10개: `/shorts`, `/blog`, `/cafe-seo`, ...
- 운영 3개: `/batch` (일괄생성), `/analyze` (키워드분석), `/report` (리포트)
- 개발/배포 5개: `/deploy`, `/verify`, `/restart`, `/code-review`, `/debug`

### 훅 구조 (4유형)

| 유형 | 위치 | 내용 |
|------|------|------|
| **Pre 훅** | settings.json + 파이프라인 내부 | 코드 수정 전 문법검사, 생성 전 중복체크, 입력값 검증 |
| **Post 훅** | settings.json + 파이프라인 내부 | 코드 수정 후 문법검사, 생성 후 품질검수 자동 호출 |
| **Stop 훅** | 파이프라인 내부 | 3회 리비전 실패→중단, 금칙어 3개+→중단, API에러 3회→중단 |
| **Notification 훅** | 파이프라인 내부 | 단계별 진행상황, 품질점수, 완료 보고 |

### 피드백 루프

```
script-writer (생성)
    ↓
script-reviewer (1차: 규칙체크 + 2차: AI평가)
    ↓
PASS (AI 7점 이상) → 다음 단계
FAIL (AI 7점 미만) → 피드백 + 재생성 (최대 3회)
    ↓
3회 실패 → STOP → 최고점 결과 선택 + 사장에게 수동 검토 요청
```

채널별 규칙 기반 품질 기준 예시:
- 블로그: 2200자+, 키워드 8회+, 소제목 4개+
- 숏츠: 300~800자, 첫 문장 훅 필수, CTA 필수
- 카페SEO: 800~1500자, 댓글 3개+, 광고 티 안 나야 함

### 하네스 (4요소)

| 요소 | 구현 |
|------|------|
| 컨텍스트 파일 | CLAUDE.md에 강제 규칙 (품질기준, 학습루프, 라우팅, 도구경계) |
| CI/CD 게이트 | Pre/PostToolUse 훅(문법검사), 품질검수 자동 실행 |
| 도구 경계 | 에이전트별 읽기전용/생성전용/쓰기전용 명시 |
| 피드백 루프 | 실수→CLAUDE.md 학습루프에 규칙 추가→같은 실수 구조적 불가능 |

### 마이그레이션 로드맵

- Phase 1: 숏츠 파이프라인 먼저 구축 (파일럿 검증)
- Phase 2: 블로그 + 나머지 채널 확장
- Phase 3: 디스코드 봇 연결 + 스케줄 자동실행
- Phase 4: 하네스 고도화 (품질점수 추적, 자동 학습)

### 기존 시스템과의 관계

server.py + dashboard.html은 **그대로 유지**. 파이프라인 에이전트가 기존 API를 "도구"로 호출하는 래핑 구조. 기존 웹 대시보드도 계속 사용 가능.

## 검증 요청사항

이 아키텍처에 대해 다음을 검증해주세요:

### 1. 구조적 타당성
- 2단 계층(파이프라인→서브에이전트)이 적절한가? 3단이 필요한 경우가 있는가?
- 총괄 오케스트레이터를 별도 에이전트로 만들지 않고 CLAUDE.md + 메인 세션이 담당하는 것이 맞는가?
- 서브에이전트 15개의 분리 단위가 적절한가? 너무 잘게 쪼갠 건 아닌가? 합쳐야 할 것은?

### 2. 피드백 루프
- 규칙 체크 → AI 평가 → 재생성 (최대 3회) 구조가 효과적인가?
- AI 점수 7.0을 기준으로 잡은 것이 적절한가?
- script-writer와 script-reviewer를 분리한 것이 맞는가? 한 에이전트가 생성+자체 평가하는 것보다 나은가?

### 3. 하네스
- CLAUDE.md 학습 루프(실수→규칙추가→시스템강화) 방식이 실제로 효과적인가?
- 도구 경계(읽기전용/쓰기전용)가 충분한 제약인가?
- 놓친 하네스 요소가 있는가?

### 4. 확장성
- 10개 채널이 모두 같은 패턴(파이프라인→서브에이전트)인데, 채널 특성이 매우 다를 경우 문제가 생기지 않는가?
- 디스코드 봇 연결 시 이 구조가 자연스럽게 확장되는가?
- 새 채널 추가 시 정말 "에이전트 .md 파일 추가"만으로 되는가?

### 5. 리스크
- 이 구조의 가장 큰 약점은 무엇인가?
- Claude API 비용이 피드백 루프 때문에 폭증하지 않는가?
- 파이프라인이 서브에이전트를 순차 spawn하면 속도가 너무 느리지 않는가?
- 실패 시나리오(API 다운, 토큰 초과 등)에 대한 대비가 충분한가?

### 6. 대안 제시
- 이 설계보다 더 나은 대안이 있다면 구체적으로 제시해주세요.
- 특히 "1인 이커머스 사업자가 AI 직원 팀을 구축한다"는 목표에 더 적합한 접근이 있는가?
