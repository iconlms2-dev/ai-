# 에이전틱 아키텍처 설계 검증 요청 (v2 — 1차 리뷰 반영)

## 내 상황

나는 이커머스(건강기능식품) 1인 사업자이고, 마케팅 자동화 프로그램을 직접 만들어 쓰고 있다. 현재 Python FastAPI 백엔드(7,700줄) + HTML 프론트엔드(7,089줄) 모놀리식 구조이고, 네이버 블로그/카페/지식인, 유튜브, 틱톡, 쓰레드 등 10개 채널에 콘텐츠를 자동 생성하는 시스템이다.

## 내 목표

**최종 목표**: 디스코드/슬랙으로 AI 직원들에게 지시하고, 결과물만 받아서 최종 판단/승인하는 구조. 진짜 회사의 사장처럼 운영하고 싶다.

- AI가 각 분야의 전문가처럼 동작
- 지시 → AI가 알아서 실행 → 결과 보고 → 내가 승인
- 콘텐츠 퀄리티가 최우선 (피드백 루프로 자동 개선)
- 개발도 AI가 직접 수행 (코드 수정, 테스트, 배포)
- 실수가 반복 불가능한 구조 (하네스)

## 현재 기술 스택

- **백엔드**: Python, FastAPI (port 8000)
- **프론트엔드**: Vanilla HTML/CSS/JS
- **AI**: Claude API (Anthropic), Gemini API (Google)
- **연동**: Notion API, 네이버 검색광고 API, Cafe24, ElevenLabs TTS, Meta Threads
- **개발 도구**: Claude Code (Anthropic의 AI 코딩 에이전트)

## 적용하려는 에이전틱 엔지니어링 5요소

1. **Agent & Sub-Agent**: 멀티 에이전트 오케스트레이션
2. **Context Engineering**: 3레이어 컨텍스트 (시스템/대화/도구)
3. **Tool & MCP**: 도구 계층 구조
4. **Hook & Skill**: 자동 인터셉터 + 베스트 프랙티스 문서
5. **Planning & HITL**: 계획 수립 + 핵심 포인트에서 사람 확인

그리고 **하네스**(실패가 구조적으로 반복 불가능하도록 구조 자체를 고치는 것):
- 컨텍스트 파일 (CLAUDE.md = 강제 규칙)
- CI/CD 게이트 (자동 검사)
- 명시적 도구 경계 (읽기전용/쓰기전용)
- 지속적 피드백 루프 (실수→규칙추가→하네스강화)

## 1차 리뷰 결과 요약

GPT-4o에게 v1 설계를 검증받았고, 주요 지적은:

1. **"프롬프트가 많은 긴 직렬 파이프라인이 될 가능성"** — 에이전트라고 부르지만 실제로는 프롬프트 체인에 가깝다
2. **"상태 관리 부재"** — 세션 끊기면 작업 진행 상태가 날아감, 중복 방지 없음
3. **"에이전트 15개는 과하다, 5개로 줄여라"** — 관리 부담
4. **"총점 7.0 기준은 흔들린다"** — 항목별 문턱 기준이 낫다
5. **"전체 재생성 3회는 비효율"** — 원인별 부분 수정이 낫다
6. **"CLAUDE.md 학습 루프가 비대해질 것"** — 규칙 분류 필요
7. **"AI가 흐름 관리하면 안 된다, 코드가 관리해야"** — 워크플로 엔진 추천
8. **"notion-saver 등은 에이전트가 아니라 도구 어댑터"**

## 1차 리뷰에서 수용한 것

| 지적 | 수용 여부 | 이유 |
|------|----------|------|
| 상태 관리 추가 (작업 저장소) | **수용** | 세션 끊겨도 이어가기, 중복 방지 필수 |
| 총점→항목별 문턱 기준 | **수용** | 채널마다 기준 다르고, 부분 수정 가능해짐 |
| 전체 재생성→원인별 부분 수정 | **수용** | 비용 절감 + 잘 쓴 부분 보존 |
| CLAUDE.md 규칙 분류 (영구/채널/실험/폐기) | **수용** | 비대화 방지 |
| 중복 방지 키, 멱등성 | **수용** | 운영 안정성 |
| 출력 스키마 검사 | **수용** | 단계 간 데이터 깨짐 방지 |
| notion-saver 등을 도구 어댑터로 전환 | **수용** | 에이전트가 아니라 도구가 맞음 |

## 1차 리뷰에서 거부한 것

| 지적 | 거부 이유 |
|------|----------|
| **에이전트 15개→5개 축소** | 개발/관리를 AI가 하므로 관리 부담이 사람에게 없음. 오히려 15개가 각자 컨텍스트가 짧아서 품질이 높고, 새 채널 추가 시 기존 에이전트를 안 건드림 |
| **AI는 작업만, 흐름은 코드가 관리** | 이러면 "AI 직원 팀"이 아니라 "자동화 파이프라인"이 됨. AI가 판단+실행+보고를 해야 "직원"임. 단, 상태 저장은 코드로 보강 |
| **풀 워크플로 엔진 먼저 구축** | 분석 마비 위험. 숏츠 파일럿 먼저 돌리고 필요한 만큼만 추가 |

## v2 수정 설계

### 전체 구조

```
[나 (사장)] ─── 디스코드/슬랙 또는 Claude Code CLI
    │
    ▼
[메인 세션 = Claude Code]
  CLAUDE.md(하네스) = 강제 규칙
  운영 모드: 파이프라인 에이전트 spawn → 콘텐츠 생성
  개발 모드: 직접 코드 수정 + 테스트 + 배포
    │
    ├→ [shorts-pipeline] → 서브에이전트들 spawn
    ├→ [blog-pipeline]   → 서브에이전트들 spawn
    ├→ ... (10개 채널)
    │
    └→ [작업 상태 저장소] ← ★v2 추가
         job_id, step, status, artifacts
         (JSON 파일. 세션 끊겨도 이어가기 가능)
```

### 에이전트 계층 (2단, 15개 서브에이전트 유지)

**L1 파이프라인 에이전트 (10개)** — 채널별 오케스트레이터:

| 파이프라인 | 단계 |
|-----------|------|
| shorts-pipeline | 벤치마킹 → 전략 → 대본 → 품질검수(→부분수정) → TTS → 저장 |
| blog-pipeline | 키워드분석 → 전략 → 제목 → 본문 → 품질검수(→부분수정) → 저장 |
| cafe-seo-pipeline | 키워드분석 → 전략 → 제목+본문 → 댓글 → 품질검수 → 저장 |
| cafe-viral-pipeline | 1단계(관심) → 2단계(문제) → 3단계(솔루션) → 품질검수 → 저장 |
| jisikin-pipeline | 키워드분석 → 질문생성 → 답변생성 → 품질검수 → 저장 |
| youtube-pipeline | 영상검색 → 영상분석 → 댓글생성 → 품질검수 → (자동게시) → 저장 |
| tiktok-pipeline | 키워드분석 → 스크립트 → 품질검수 → 저장 |
| community-pipeline | 전략수립 → 게시글 → 댓글 → 품질검수 → 저장 |
| powercontent-pipeline | 레퍼런스수집 → 분석 → 광고카피 → 본문 → 품질검수 → 저장 |
| threads-pipeline | 레퍼런스수집 → 유형선택 → 콘텐츠 → 품질검수 → 발행 → 저장 |

**L2 서브에이전트 (15개)** — 전문 작업자:

| 서브에이전트 | 역할 | 도구 경계 | 사용 채널 수 |
|-------------|------|----------|-------------|
| data-researcher | 외부 데이터 수집 (URL, 검색) | 읽기전용 | 4 |
| pattern-extractor | 패턴/팩트 추출 | 읽기전용 | 2 |
| keyword-analyzer | SERP분석 + 경쟁강도 | 읽기전용 | 4 |
| video-analyst | 영상 정보 분석 | 읽기전용 | 2 |
| strategist | 전략/컨셉 수립 | 생성전용 | 5 |
| hook-designer | 훅/CTR 설계 | 생성전용 | 2 |
| title-generator | 제목 생성 | 생성전용 | 3 |
| script-writer | 본문/대본 작성 | 생성전용 | 9 |
| comment-writer | 댓글 생성 | 생성전용 | 4 |
| script-reviewer | 품질 검수 (규칙+AI) | 읽기전용 | 10 |
| forbidden-checker | 금칙어 체크 | 읽기전용 | 3 |
| tts-converter | TTS+자막 변환 | 도구어댑터 | 1 |
| notion-saver | Notion 저장 | 도구어댑터 | 10 |
| youtube-poster | 유튜브 게시 | 도구어댑터(HITL) | 1 |
| threads-publisher | 쓰레드 발행 | 도구어댑터(HITL) | 1 |

**15개 유지 이유:**
- 개발/관리를 AI가 하므로 사람의 관리 부담 없음
- 각 에이전트가 하나만 잘하면 됨 → 컨텍스트가 짧아서 품질 높음
- 새 채널 추가 시 기존 에이전트 안 건드림 (확장성)
- tts-converter, notion-saver, youtube-poster, threads-publisher는 "도구 어댑터"로 분류 (AI 판단 안 함, 실행만)

**채널별 차이 대응:**
- 채널마다 전용 매뉴얼 파일 (channel-manuals/*.md)
- script-writer가 호출될 때 해당 채널 매뉴얼만 컨텍스트로 전달
- 10개 채널 규칙을 한 번에 안 줌 → 컨텍스트 오염 방지

### 훅 구조 (4유형)

| 유형 | 위치 | 내용 |
|------|------|------|
| Pre 훅 | settings.json + 파이프라인 내부 | 코드 수정 전 문법검사, 생성 전 중복체크, 입력값 검증 |
| Post 훅 | settings.json + 파이프라인 내부 | 코드 수정 후 문법검사, 생성 후 품질검수 자동 호출 |
| Stop 훅 | 파이프라인 내부 | 3회 부분수정 실패→중단, 금칙어 3개+→중단, API에러 3회→중단 |
| Notification 훅 | 파이프라인 내부 | 단계별 진행상황, 품질점수, 완료 보고 |

### 피드백 루프 (★v2 변경: 전체 재생성→원인별 부분 수정)

**v1:**
```
생성 → 검수 → FAIL → 전체 재생성 (최대 3회)
```

**v2:**
```
생성 → 검수 → FAIL
  → 실패 원인 분류:
    - 규칙 실패 (키워드 부족, 글자수 미달) → 해당 부분만 수정 지시
    - 구조 실패 (흐름 깨짐, 단계 누락) → 해당 섹션 재작성
    - 톤 실패 (광고 티, 부자연스러움) → 톤 조정 재작성
    - 전략 실패 (방향 자체가 틀림) → 상위 단계(strategist)로 되돌림
  → 이전 최고안과 비교 → 나아졌을 때만 교체
  → 최대 3회 부분 수정 후에도 FAIL → STOP → 사장에게 수동 검토 요청
```

### 품질 기준 (★v2 변경: 총점→항목별 문턱)

**v1:** AI 점수 7.0 이상이면 PASS

**v2:** 항목별 하한선. 하나라도 미달이면 FAIL + 해당 항목만 수정.

예시 (블로그):
| 항목 | 하한선 | 체크 방식 |
|------|--------|----------|
| 글자수 | 2,200자 이상 | 코드 (규칙) |
| 키워드 포함 | 8회 이상 | 코드 (규칙) |
| 소제목 | 4개 이상 | 코드 (규칙) |
| 금칙어 | 0개 | 코드 (규칙) |
| 자연스러움 | 7/10 이상 | AI 평가 |
| 정보성 | 7/10 이상 | AI 평가 |
| 채널 적합도 | 7/10 이상 | AI 평가 |
| 설득력 | 6/10 이상 | AI 평가 |

→ 규칙 항목은 코드로 검사 (AI 안 씀 = 비용 절감)
→ AI 평가는 코드 검사 통과한 후에만 실행 (불필요한 AI 호출 방지)

### 하네스 (4요소)

| 요소 | 구현 | ★v2 변경 |
|------|------|---------|
| 컨텍스트 파일 | CLAUDE.md | 규칙을 영구/채널/실험/폐기로 분류 |
| CI/CD 게이트 | Pre/PostToolUse 훅 + 자동 품질검수 | 규칙 검사는 코드로, AI 평가는 최소화 |
| 도구 경계 | 에이전트별 읽기전용/생성전용/쓰기전용 | 도구 어댑터 4개 분리 (tts, notion, youtube, threads) |
| 피드백 루프 | 실수→규칙추가→하네스강화 | 회귀 테스트 추가 (새 규칙이 다른 채널 망치지 않는지) |

### 작업 상태 관리 (★v2 신규)

```json
// job_state.json 예시
{
  "jobs": [
    {
      "job_id": "shorts-20260406-001",
      "channel": "shorts",
      "status": "in_progress",
      "current_step": 3,
      "steps": [
        {"step": 1, "name": "벤치마킹", "status": "완료", "artifact": "bench_result.json"},
        {"step": 2, "name": "전략", "status": "완료", "artifact": "strategy.json"},
        {"step": 3, "name": "대본", "status": "진행중", "revision_count": 1},
        {"step": 4, "name": "TTS", "status": "대기"},
        {"step": 5, "name": "저장", "status": "대기"}
      ],
      "created_at": "2026-04-06T10:30:00",
      "keyword": "다이어트 보조제",
      "dedup_key": "shorts:다이어트보조제:20260406"
    }
  ]
}
```

- 세션 끊겨도 current_step부터 이어서 진행
- dedup_key로 같은 콘텐츠 중복 생성 방지
- 각 step의 artifact(산출물) 파일 참조 가능

### 마이그레이션 로드맵

- Phase 1: 숏츠 파이프라인 (파일럿)
- Phase 2: 블로그 + 나머지 채널 확장
- Phase 3: 디스코드 봇 연결 + 스케줄 자동실행
- Phase 4: 하네스 고도화

### 기존 시스템과의 관계

server.py + dashboard.html은 **그대로 유지**. 파이프라인 에이전트가 기존 API를 "도구"로 호출하는 래핑 구조.

## 검증 요청사항

이전 리뷰에서 받은 피드백을 반영한 v2 설계이다. 다음을 검증해달라:

### 1. 에이전트 15개 유지 판단
- 1차 리뷰에서 "5개로 줄여라"고 했지만, "개발/관리를 AI가 하므로 관리 부담이 사람에게 없다"는 이유로 15개를 유지했다. 이 판단이 타당한가?
- 15개 각각의 컨텍스트가 짧아서 품질이 높다는 논리는 맞는가?
- 그래도 합쳐야 할 에이전트가 있다면 구체적으로 어떤 것?

### 2. 상태 관리 설계
- job_state.json 방식이 1인 사업자 규모에 적절한가?
- DB가 필요한 시점은 언제인가?
- 세션 끊김 → 재개 흐름에 빠진 것이 있는가?

### 3. 피드백 루프 v2
- 원인별 부분 수정(규칙실패/구조실패/톤실패/전략실패)이 전체 재생성보다 실제로 나은가?
- "전략 실패 시 상위 단계로 되돌림"이 무한 루프에 빠질 위험은?
- 항목별 문턱 기준이 총점보다 확실히 나은가?

### 4. "AI가 흐름도 관리"하는 구조의 리스크
