# 에이전틱 아키텍처 v4 최종 검증 요청

## 내 상황

1인 이커머스(건강기능식품) 사업자. 마케팅 자동화 프로그램을 직접 만들어 쓰고 있다. Python FastAPI 백엔드(7,700줄) + HTML 프론트엔드(7,089줄). 10개 채널 콘텐츠 자동 생성 + 분석/운영/광고 기능 보유.

## 내 목표

슬랙으로 AI 직원들에게 지시 → AI가 알아서 실행 → 결과 보고 → 내가 승인. 진짜 회사 사장처럼. 콘텐츠 퀄리티 최우선. 개발도 AI가 직접 수행. 실수가 반복 불가능한 구조(하네스).

## 리뷰 이력

- v1 (6.5/10): 프롬프트 체인, 상태관리 부재
- v2 (7.8/10): 상태저장소/항목별문턱/부분수정 개선. AI흐름관리 위험 지적
- v3 (8.4/10): Planning Loop/상태전이/원칙11개 반영. "범위가 넓다, 33개 과하다" 지적

## v4 핵심 변경: 총괄 오케스트레이터 삭제, 채널별 독립 구조

### v3→v4 변경 사항

| v3 | v4 |
|----|-----|
| 총괄 오케스트레이터(메인세션+CLAUDE.md)가 전체 관리 | **총괄 없음. 채널별 독립 오케스트레이터** |
| planning-loop-agent를 Phase 1에 포함 | **Phase 3로 이동** |
| 분석팀(4개), 운영팀(6개)을 전면 설계 | **Phase 3로 이동. 지금은 기존 대시보드 사용** |
| 에이전트 33개 총합 | **Phase 1은 숏츠만. 파이프라인 1 + 서브에이전트 3~5 + 도구어댑터 2** |

### 변경 이유

참고 프로젝트(유튜브 자동화)를 다시 보니, 그 프로젝트도 유튜브 1개 채널에 대한 독립 오케스트레이터였다. 총괄이 없었다. 우리도 각 채널을 독립적으로 만들고, 사용자가 /shorts, /blog 등으로 직접 실행하면 된다. 총괄 AI가 "어느 채널을 실행할지" 판단할 필요가 없다. 그건 사람(나)이 하거나, 나중에 슬랙 봇이 하면 된다.

## v4 설계

### 핵심 구조

```
[나 (사장)]
    │
    ├─ /shorts   → [숏츠 오케스트레이터] → 서브에이전트들 → 결과
    ├─ /blog     → [블로그 오케스트레이터] → 서브에이전트들 → 결과
    ├─ /cafe-seo → [카페SEO 오케스트레이터] → 서브에이전트들 → 결과
    ├─ ... (각 채널 독립, 총괄 없음)
    │
    └─ 슬랙 (Phase 3):
       "블로그 5개 숏츠 3개" → 봇이 /blog 5번, /shorts 3번 호출
```

각 채널 = 참고 프로젝트(유튜브 자동화)와 동일한 구조.
채널 간 공유되는 것: 서브에이전트(script-writer 등), CLAUDE.md, job_state.json

### 채널별 오케스트레이터 공통 패턴

```
{채널}-pipeline.md
  ├─ 조사/분석 단계
  ├─ 전략 수립 단계
  ├─ 콘텐츠 생성 단계
  ├─ 품질 검수 ←→ 부분 수정 루프
  ├─ 저장/발행 단계
  └─ 보고
```

### 서브에이전트 (채널 간 재사용, 10개)

| 에이전트 | 역할 | 도구 경계 | 사용 채널 수 |
|---------|------|----------|-------------|
| data-researcher | 외부 데이터 수집 | 읽기전용 | 4 |
| pattern-extractor | 패턴/팩트 추출 | 읽기전용 | 2 |
| keyword-analyzer | SERP+경쟁강도 | 읽기전용 | 4 |
| video-analyst | 영상 정보 분석 | 읽기전용 | 2 |
| strategist | 전략/컨셉 수립 | 생성전용 | 5 |
| hook-designer | 훅/CTR 설계 | 생성전용 | 2 |
| title-generator | 제목 생성 | 생성전용 | 3 |
| script-writer | 본문/대본 작성 | 생성전용 | 9 |
| comment-writer | 댓글 생성 | 생성전용 | 4 |
| script-reviewer | 품질 검수 (규칙+AI) | 읽기전용 | 10 |

### 도구 어댑터 (4개, 에이전트 아님)
tts-converter, notion-saver, youtube-poster(HITL), threads-publisher(HITL)

### 코드 검사기 (2개, 에이전트 아님)
forbidden-checker, rule-validator

### CLAUDE.md — 60줄 이하 (지도)

```
금지: .env 커밋, API키 하드코딩, 검수 미통과 저장, 승인 없이 발행
상태 전이: draft→under_review→revision→approved→publish_ready→published
품질 게이트: 규칙검사(코드)→AI평가→항목별 하한선. 부분수정 3회, 전략되돌림 1회.
컨텍스트: 해당 채널 매뉴얼만 로드.
학습 루프: 실수→규칙 추가 테이블
```

### 피드백 루프 (4레벨)

- Level 0 (코드): PostToolUse훅 → 문법검사 → 자체수정
- Level 1 (콘텐츠): rule-validator(코드) → script-reviewer(AI) → 부분수정
- Level 2 (파이프라인): 작업 완료 후 교훈 기록
- Level 3 (시스템, Phase 3): 성과→리포트→재계획→다음 주 반영

### 상태 관리

job_state.json + 상태 전이표(코드 강제)
```
draft → under_review → revision → under_review
under_review → approved → publish_ready → published
건너뛰기 불가, 역행 불가, 승인 없이 발행 불가
```

### 훅

- 시스템: PostToolUse (server.py 수정 후 문법검사)
- 파이프라인: PRE(중복체크, 상태확인), POST(검수자동), STOP(3회초과→HITL), NOTIFY(진행보고)

### 채널 매뉴얼 (channel-manuals/*.md)

필수 5섹션: 목적, 금지, 좋은 예시 3개, 품질 기준표, 실패 패턴

### 컨텍스트 전략

- CLAUDE.md(60줄): 모든 에이전트
- agent.md: 해당 에이전트만
- channel-manual: 해당 채널 작업 시에만
- 다른 채널 매뉴얼 안 읽음
- 안 바뀌는 정보 앞 고정, 바뀌는 정보 뒤

### 프로젝트 구조

한 프로젝트(안티그래비티/)에 전부. 에이전트가 server.py에 접근해야 하므로 분리 불가.

```
안티그래비티/
  .claude/
    agents/          ← 파이프라인 + 서브에이전트
    commands/        ← /shorts, /blog 등 스킬
    channel-manuals/ ← 채널별 매뉴얼
    settings.local.json
  server.py
  dashboard.html
  CLAUDE.md
  job_state.json
```

### 마이그레이션

- Phase 1: 숏츠 파일럿 (파이프라인 1 + 서브에이전트 + 매뉴얼 + 상태관리 + 훅)
- Phase 2: 나머지 9개 채널 확장 (서브에이전트 재사용)
- Phase 3: 슬랙 봇 + 분석/운영 에이전트 + Planning Loop + 스케줄

### Phase 1 성공 기준
1. /shorts 실행 → 숏츠 대본이 끝까지 나오는가
2. 검수 → 부분수정 → 재검수 루프가 작동하는가
3. 상태 전이가 강제되는가 (검수 없이 저장 불가)
4. 세션 끊겨도 이어가기가 되는가
5. 프롬프트 교체가 결과물에 즉시 반영되는가

### Phase 1에서 만드는 파일 (이것만)
1. CLAUDE.md (60줄)
2. shorts-pipeline.md
3. strategist.md
4. script-writer.md
5. script-reviewer.md
6. shorts-manual.md
7. /shorts 커맨드
8. rule-validator (코드)
9. job_state.json + 상태 전이
10. PostToolUse 훅

## 검증 요청

### 1. 총괄 오케스트레이터 삭제 판단
- 총괄 없이 채널별 독립 오케스트레이터로 가는 게 맞는가?
- 슬랙 봇이 나중에 라우팅 역할을 하면 충분한가?
- 채널 간 조율이 필요한 상황(예: 같은 키워드로 블로그+카페 동시 생성)에서 총괄 없이 문제가 생기지 않는가?

### 2. 참고 프로젝트와의 정합성
- 참고 프로젝트(유튜브 자동화)는 1채널 독립 오케스트레이터 구조인데, 이걸 10채널로 복제하는 접근이 적절한가?
- 채널 간 서브에이전트 공유(script-writer 등)가 독립 구조와 충돌하지 않는가?

### 3. Phase 1 범위
- 숏츠 파일럿에 만드는 파일 10개가 적절한가?
- 빠진 것이 있는가? 과한 것이 있는가?
- 이 범위로 시작해서 핵심 가설(AI가 승인 가능한 숏츠를 안정적으로 만드는가)을 검증할 수 있는가?

### 4. v3 대비 v4 개선도
- v3(8.4/10) 대비 v4는 몇 점인가?
- 총괄 삭제와 범위 축소가 올바른 방향인가?
- 이 설계대로 Phase 1 구현을 시작해도 되는가?

### 5. 남은 약점
- 이 설계의 가장 큰 약점은 무엇인가?
- 구현 시작 전에 반드시 추가하거나 바꿔야 할 것이 있는가?

솔직하게 검증해줘. 이번이 구현 시작 전 마지막 리뷰다.
