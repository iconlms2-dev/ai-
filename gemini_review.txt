# 코드 리뷰 요청: Meta Threads 자동화 — Gemini 피드백 반영 + 3가지 SL 추가

## 변경 배경/목적
이전 리뷰에서 받은 Gemini 피드백 3건을 반영하고, Threads 물길글의 셀링 로직(SL)을 3가지로 세분화했습니다.

## 변경 내용 요약

### 1. Gemini 피드백 반영 (server.py)

**토큰 자동 갱신**
- `_threads_refresh_token(acc)` 함수 신규 추가
- Threads 장기 토큰 만료 15일 전 자동으로 `/refresh_access_token` API 호출
- 스케줄러 루프(`_threads_scheduler_loop`)에서 60분마다 전체 계정 토큰 만료 체크
- 갱신 성공 시 threads_accounts.json에 새 토큰 + 만료일 저장

**프롬프트 톤 수정 (3개 프롬프트 전부)**
- Threads 플랫폼 특성 반영: 짧은 단문(1~5문장), 줄임말/구어체, 혼잣말 톤
- 200자 이내 권장 (기존 500자에서 축소)
- 블로그/카페식 정리된 글 금지 → 생각을 툭 던지는 느낌
- 이모티콘 0~1개, 해시태그 거의 안 씀
- 일상글: "~인 듯", "~임", "~ㅋㅋ" 등 다양한 문장 끝
- 댓글: 30자 이내 1문장으로 축소, "정보 감사합니다" 같은 정중한 톤 금지

**SSE ClientDisconnect 처리**
- `threads_generate` 엔드포인트: 일상글/물길글 루프 매 반복마다 `await request.is_disconnected()` 체크
- 클라이언트 탭 닫으면 즉시 생성 중단 → Claude API 호출 낭비 방지
- `threads_crawl_reference` 엔드포인트: 참조계정 크롤링 중에도 disconnect 체크 → Playwright 리소스 낭비 방지

### 2. 물길글 3가지 SL(셀링 로직) 추가

`_build_threads_traffic_prompt()`에 `selling_logic` 파라미터 추가. 3가지 분기:

**정보 셔플식 (shuffle)** — 기존 방식
- 대안 3~4개 나열 + 제품 간접 노출
- 제품명 직접 언급 금지, 나만의 키워드만 1회

**연민 판매자식 (sympathy)** — 신규
- 불쌍한 사연 + 대놓고 홍보
- "퇴사하고 올인했다", "3년 걸렸다" 식의 동정 유발
- 제품명/브랜드 직접 언급 OK
- 구걸 느낌 아니고 "진심을 담았다" 느낌

**후기 신뢰식 (review)** — 신규
- 전문가 포지셔닝 → 비교 대상 언급 → 담백한 후기
- 페르소나 직업/관심사가 전문성 근거
- "A도 써봤고 B도 써봤는데" → 구체적 수치/기간 포함
- 제품명/브랜드 직접 언급 OK

**공통: 모든 SL에 첫 줄 어그로 필수**
- 부정 편향, 숫자 충격, 권위, 호기심, 공감 패턴 중 하나 사용
- 첫 문장에서 스크롤 멈추게 하는 것이 핵심

### 3. 프론트엔드 (dashboard.html)
- 물길글 탭에 "셀링 로직 (SL)" 드롭다운 추가 (정보 셔플식/연민 판매자식/후기 신뢰식)
- 기존 2열 그리드 → 3열로 변경 (계정/SL/생성수)
- JS에서 `selling_logic` 값을 API 호출 시 전달

## 리뷰 요청 사항
1. **3가지 SL 프롬프트 품질**: 각 셀링 로직의 프롬프트가 실제 Threads 톤에 적합한지? 특히 연민 판매자식과 후기 신뢰식이 너무 광고처럼 느껴지지 않는지
2. **어그로 첫 줄 패턴**: 5가지 어그로 패턴이 충분한지, 추가할 패턴이 있는지
3. **토큰 갱신 로직**: 60분마다 체크 → 15일 전 갱신 방식이 적절한지, edge case (갱신 실패 시 재시도 등) 처리가 충분한지
4. **ClientDisconnect**: `request.is_disconnected()` 체크 위치가 적절한지 (현재 루프 시작점에서만 체크 — Claude API 호출 중에는 감지 못함)
5. **SL 간 전환**: 같은 계정에서 셔플식 → 연민식 → 후기식을 섞어 쓸 때 글 톤의 일관성 문제가 있는지
6. **연민 판매자식의 리스크**: Threads에서 동정 마케팅이 역효과를 낼 가능성 (반감 유발) — 프롬프트에 방어 규칙이 충분한지
